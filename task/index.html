Prep & Setup

Prerequisites and local environment setup for running the project from scratch.

1. Install NVM

macOS / Linux

curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.1/install.sh | bash

Reload shell

source ~/.bashrc
# or
source ~/.zshrc

Verify

nvm --version

2. Install Node.js via NVM

nvm install 20
nvm use 20
nvm alias default 20

Verify

node -v
npm -v
Node 20 reaches EoL April 2026. Plan to migrate to Node 24.

3. Install Docker

macOS (Docker Desktop)

brew install --cask docker

Open Docker Desktop once to complete setup and accept the license.

Linux (Docker Engine)

curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER

Verify

docker --version
docker compose version

4. Clone & Install

git clone <repo-url>
cd <project-dir>
npm install

5. Environment Variables

cp .env.example .env

Required variables:

JWT_SECRET=your_jwt_secret
MONGODB_URI=mongodb://localhost:27017/real-estate
CLOUDINARY_NAME=your_cloudinary_name
CLOUDINARY_API_KEY=your_cloudinary_api_key
CLOUDINARY_API_SECRET=your_cloudinary_api_secret
SENDGRID_API_KEY=your_sendgrid_key

6. Install PM2

npm install -g pm2

7. Run the Project

npm run dev

Uses PM2 to run both backend (port 4000) and frontend (port 3000) with auto-reload.

  • Frontend: http://localhost:3000
  • Backend API: http://localhost:4000
pm2 logs          # View logs
npm run dev:stop  # Stop

Quick Verify Checklist

nvm --version          # NVM installed
node -v                # Node 20.x
npm -v                 # npm available
docker --version       # Docker installed
docker compose version # Compose v2 available
pm2 --version          # PM2 installed
npm run dev            # Project starts without errors

Setup Notes

Recommendation: Update Node 20 to 24 as 20 will be EoL on April 2026.

Startup Script Migration

The original startup script was problematic:

npm start  вҶ’  "node server/server.js | next dev"

Single pipe, no process isolation, no restart handling.

Problems

  • Single pipe failure вҖ” if server fails, Next.js continues orphaned
  • No auto-restart вҖ” crashes require manual restart
  • No log isolation вҖ” mixed stdout/stderr
  • No process management вҖ” can't monitor resources or graceful shutdown

Solution: PM2 Ecosystem Config

// ecosystem.config.js
module.exports = {
  apps: [
    {
      name: 'backend',
      script: 'server/server.js',
      env: { NODE_ENV: 'development', PORT: 4000 },
      error_file: 'logs/backend-error.log',
      out_file: 'logs/backend-out.log',
    },
    {
      name: 'frontend',
      script: 'node_modules/.bin/next',
      args: 'dev',
      env: { NODE_ENV: 'development', PORT: 3000 },
      error_file: 'logs/frontend-error.log',
      out_file: 'logs/frontend-out.log',
    }
  ]
};

Updated package.json scripts

{
  "dev": "pm2-dev ecosystem.config.js",
  "dev:pm2": "pm2 start ecosystem.config.js",
  "dev:stop": "pm2 stop all",
  "dev:restart": "pm2 restart all",
  "dev:logs": "pm2 logs",
  "build": "next build",
  "start": "pm2 start ecosystem.config.js --no-daemon"
}

рҹ”’ Security Incident: C2 Attack Vector Removed

File removed: server/middlewares/validator/errorHandler.js
Classification: Command and Control (C2) attack вҖ” remote code execution vector disguised as error handling middleware.

Attack Mechanism

  • Exfiltration вҖ” decoded base64 strings from RUNTIME_CONFIG_* env vars to get API endpoint + auth header
  • Remote Fetch вҖ” HTTP GET to attacker-controlled endpoint, response stored in globalConfig
  • Trigger вҖ” errorTimeHandler() auto-executes during error handling

Required Actions

  • File deleted from codebase вң…
  • Check git history for similar obfuscated patterns
  • Audit environment variables вҖ” any RUNTIME_CONFIG_* keys are suspicious
  • Review git logs to identify who added this file
  • Scan for other middleware files with hidden HTTP requests

Terraform

Directory Structure

terraform/ в”ңв”Җв”Җ run-tf.sh # Wrapper script в”ңв”Җв”Җ modules/ в”Ӯ в”ңв”Җв”Җ network/ # VPC, subnets, IGW, NAT, route tables в”Ӯ в”Ӯ в”ңв”Җв”Җ main.tf в”Ӯ в”Ӯ в”ңв”Җв”Җ variables.tf в”Ӯ в”Ӯ в””в”Җв”Җ outputs.tf в”Ӯ в””в”Җв”Җ ec2/ # EC2 instance, security group, user data в”Ӯ в”ңв”Җв”Җ main.tf в”Ӯ в”ңв”Җв”Җ variables.tf в”Ӯ в””в”Җв”Җ outputs.tf в””в”Җв”Җ env/ в”ңв”Җв”Җ dev/ # Dev environment config в””в”Җв”Җ prod/ # Prod environment config

How It Works

Modules are reusable building blocks. Environments consume them with different variable values. Each environment has its own state file in S3, so dev and prod are completely isolated.

env/dev/main.tf в”Җв”Җusesв”Җв”Җв–¶ modules/network в”Җв”Җusesв”Җв”Җв–¶ modules/ec2 env/prod/main.tf в”Җв”Җusesв”Җв”Җв–¶ modules/network в”Җв”Җusesв”Җв”Җв–¶ modules/ec2

Module: network

Creates the full VPC networking layer: VPC with DNS, Internet Gateway, public subnets (auto-assign public IP), private subnets, route tables, and NAT Gateways.

NAT Gateway Modes

ModeBehaviorCostUse Case

Module: ec2

Creates an EC2 instance with a security group (SSH port 22 only, all outbound). Encrypted gp3 root volume. Default user data installs Docker + Compose v2.

Remote State (S3 Backend)

Both environments store state in the same S3 bucket under different keys. State locking uses DynamoDB.

Prerequisites (one-time)

# S3 bucket for state
aws s3 mb s3://myproject-terraform-state --region us-east-1
aws s3api put-bucket-versioning \
  --bucket myproject-terraform-state \
  --versioning-configuration Status=Enabled

# DynamoDB table for locking
aws dynamodb create-table \
  --table-name terraform-locks \
  --attribute-definitions AttributeName=LockID,AttributeType=S \
  --key-schema AttributeName=LockID,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST \
  --region us-east-1

run-tf.sh (Wrapper Script)

./run-tf.sh --dev init
./run-tf.sh --dev plan
./run-tf.sh --prod apply -auto-approve
./run-tf.sh --dev destroy -target=module.ec2
./run-tf.sh --dev show

Common Workflow

# 1. Initialize
./run-tf.sh --dev init

# 2. Review
./run-tf.sh --dev plan

# 3. Apply
./run-tf.sh --dev apply

# 4. Get EC2 public IP
./run-tf.sh --dev output ec2_public_ip

# 5. SSH in
ssh -i ~/.ssh/my-key.pem ec2-user@<public-ip>

# 6. Tear down
./run-tf.sh --dev destroy

Customization

  • Instance type вҖ” edit instance_type in terraform.tfvars
  • More AZs вҖ” add entries to azs, public_subnet_cidrs, private_subnet_cidrs
  • Restrict SSH вҖ” set ssh_cidr_blocks to your IP
  • Custom bootstrap вҖ” pass user_data to the ec2 module
  • Switch AMI вҖ” update ami_id in terraform.tfvars

Fluentd (Log Forwarder)

Collects logs via two methods:

  • Docker logging driver вҖ” app stdout/stderr sent directly to Fluentd
  • Shared volume вҖ” tails /app/logs/*.log from the app-logs volume

Metadata injected: hostname, environment: production.

Default output is stdout (JSON). Swap by uncommenting a pre-configured output in observability/fluentd/fluent.conf:

  • Forward to a Fluentd/Fluent Bit aggregator
  • Elasticsearch / OpenSearch
  • CloudWatch Logs
docker logs fluentd  # View collected logs

Prometheus (Metrics)

Scrape targets:

  • app:4000/metrics вҖ” application metrics (15s interval)
  • localhost:9090 вҖ” Prometheus self-monitoring

Retention: 15 days. Storage: persistent prometheus-data volume.

Prerequisite: expose /metrics

npm install prom-client
const { collectDefaultMetrics, register } = require('prom-client');
collectDefaultMetrics();

app.get('/metrics', async (req, res) => {
  res.set('Content-Type', register.contentType);
  res.end(await register.metrics());
});

docker compose -f docker-compose.prod.yml restart fluentd
